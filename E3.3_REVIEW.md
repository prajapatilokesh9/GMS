## 2. Status Transition Review

### Allowed Transitions

```
scheduled ──→ checked_in ──→ completed
scheduled ──→ cancelled
```

- `scheduled → checked_in`: POST `/:id/check-in` sets `started_at` and status
- `checked_in → completed`: POST `/:id/complete` sets `completed_at` and status
- `scheduled → cancelled`: POST `/:id/cancel` sets `cancelled_at` and status

### Rejected Transitions

| Transition | Result |
|-----------|--------|
| `checked_in → checked_in` | 409 Conflict |
| `completed → any` | 409 Conflict |
| `cancelled → any` | 409 Conflict |
| Partial PATCH to change `status` directly | Not possible — `status` excluded from `updatePtSessionSchema` |

### Error Handling

- **409 Conflict** — invalid status transition (body: `{ "error": "Session cannot be checked in from current status" }`)
- **404 Not Found** — non-existent or cross-tenant session ID
- **401 Unauthorized** — missing or invalid auth token
- **403 Forbidden** — missing required permission
- **400 Bad Request** — Zod validation failure (missing fields, invalid UUIDs, etc.)

## 3. Security Review

### RBAC Coverage

- **All 7 routes** protected by `authenticate` + `loadPermissions` middleware
- **Read operations** (list, getById): require `session:read`
- **Write operations** (create, update, check-in, complete, cancel): require `session:manage`
- Separation of read vs. manage matches the pattern used by E3.1 (trainers) and E3.2 (pt-packages)

### Tenant Isolation

- All service-layer queries scope to `tenantId` from the authenticated user's context
- Cross-tenant reads return **404** (avoids leaking existence of records)
- Cross-tenant writes return **404**
- 4 composite indexes include `tenant_id` as the leading column for query performance
