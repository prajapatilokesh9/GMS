## 1. Prisma Schema — `AmcContract`, `AmcContractEquipment`, `AmcClaim`

### `AmcContract`

```prisma
// ============================================================
// EQUIPMENT — Annual Maintenance Contract
// ============================================================
model AmcContract {
  id                        String    @id @default(uuid()) @db.Uuid
  tenantId                  String    @map("tenant_id") @db.Uuid
  contractNumber            String    @map("contract_number") @db.VarChar(100)
  supplierId                String?   @map("supplier_id") @db.VarChar(200)        // Loose reference — supplier/AMC provider
  supplierName              String?   @map("supplier_name") @db.VarChar(200)

  // Type & coverage
  contractType              String    @default("comprehensive") @map("contract_type") @db.VarChar(20)  // comprehensive | parts_only | labor_only | custom
  coverageDescription       String?   @map("coverage_description") @db.Text

  // Schedule
  startDate                 DateTime  @map("start_date") @db.Date
  endDate                   DateTime  @map("end_date") @db.Date
  renewalPeriodDays         Int?      @default(365) @map("renewal_period_days")   // Duration for auto-renewal

  // Billing
  contractValue             Decimal   @map("contract_value") @db.Decimal(12, 2)   // Total contract value for the period
  billingFrequency          String    @default("yearly") @map("billing_frequency") @db.VarChar(20)  // monthly | quarterly | half_yearly | yearly | one_time
  currency                  String    @default("INR") @db.VarChar(3)

  // Per-claim limits
  claimLimitPerPeriod       Int?      @default(0) @map("claim_limit_per_period")  // 0 = unlimited
  claimAmountLimit          Decimal?  @map("claim_amount_limit") @db.Decimal(12, 2) // Max payout per claim (null = no limit)
  claimAmountLimitPerPeriod Decimal?  @map("claim_amount_limit_per_period") @db.Decimal(12, 2) // Aggregate cap per period

  // Status
  status                    String    @default("draft") @db.VarChar(20)           // draft | active | expiring | expired | cancelled | renewed

  // Renewal tracking
  autoRenew                 Boolean   @default(true) @map("auto_renew")
  renewedFromId             String?   @map("renewed_from_id") @db.Uuid           // Previous contract that was renewed into this one
  renewedToId               String?   @map("renewed_to_id") @db.Uuid             // New contract created by renewal of this one
  renewalReminderSentAt     DateTime? @map("renewal_reminder_sent_at")
  cancellationReason        String?   @map("cancellation_reason") @db.VarChar(500)
  expiredAt                 DateTime? @map("expired_at")

  // Soft delete
  isActive                  Boolean   @default(true) @map("is_active")
  createdAt                 DateTime  @default(now()) @map("created_at")
  updatedAt                 DateTime  @updatedAt @map("updated_at")
  deletedAt                 DateTime? @map("deleted_at")

  tenant                    Tenant             @relation(fields: [tenantId], references: [id], onDelete: Cascade)
  coveredEquipment          AmcContractEquipment[]
  maintenanceJobs           MaintenanceJob[]

  @@index([tenantId, status])
  @@index([tenantId, contractNumber])
  @@index([tenantId, supplierId])
  @@index([tenantId, endDate])
  @@map("amc_contracts")
  @@schema("equipment")
}
```

## 2. AMC Contract Lifecycle State Machine

### States

| State | Description |
|-------|-------------|
| `draft` | Contract being created/negotiated, not yet active |
| `active` | Contract in effect, equipment covered |
| `expiring` | Within the renewal window (30 days before `endDate`); system has sent or is ready to send renewal notice |
| `expired` | Contract period ended, not renewed (terminal) |
| `cancelled` | Terminated before or during term (terminal) |
| `renewed` | Contract was renewed — a follow-on contract exists (terminal) |

### Transitions

| # | From | To | Trigger | Reason Required | Guard Condition |
|---|------|----|---------|-----------------|-----------------|
| 1 | draft | active | Contract signed/approved | No | `startDate` must not be in the past unless explicitly backdated |
| 2 | active | expiring | System detects 30-day window before `endDate` | No | Automated — cron/scheduler; `autoRenew` flag checked to decide behaviour |
| 3 | active | cancelled | Manually cancelled during term | Yes | — |
| 4 | expiring | active | Renewed (auto or manual) | No | `renewalPeriodDays` used to extend; new `endDate` computed; if auto-renew and payment fails, stays `expiring` |
| 5 | expiring | expired | Renewal period lapses | No | System sets `expiredAt = now()` |
| 6 | expired | active | Late renewal (gap is acceptable) | Yes | Reason for late renewal required; new `endDate` computed from today |
| 7 | active | renewed | Full-term complete, auto-renewal succeeded | No | System creates successor `AmcContract`, links via `renewedToId`; previous contract set to `renewed` |

**Terminal states:** `expired`, `cancelled`, `renewed` — no outgoing transitions.

### Renewal Reminder Flow

> T-30 days before `endDate`: contract transitions `active → expiring`.  
> If `autoRenew = true`:
>   - System generates a renewal payment intent for the next period
>   - On successful payment: creates successor contract, sets existing to `renewed`
>   - On payment failure: contract stays `expiring`, retry mechanism invoked
> If `autoRenew = false`:
>   - Notification sent to admin/manager
>   - Manual renewal required before `endDate`
> T-0 days: contract transitions `expiring → expired`

## 3. Renewal and Expiry Workflow

### Auto-Renewal (cron/scheduled job)

```
1. Cron job runs daily, queries AmcContract where:
     status = 'active'
     AND endDate BETWEEN now() AND now() + INTERVAL '30 days'
   Sets status = 'expiring', publishes amc.contract.expiring

2. Cron job runs daily, queries AmcContract where:
     status = 'expiring'
     AND autoRenew = true
     AND endDate >= now()
   For each:
     a. Attempt payment via billing.payments (amount = contractValue, period-matched)
     b. If payment succeeds:
        - Create new AmcContract (successor) with:
            startDate = old.endDate + 1 day
            endDate = startDate + renewalPeriodDays
            renewedFromId = old.id
        - Set old contract status = 'renewed', renewedToId = new.id
        - Copy AmcContractEquipment records to new contract
        - Publish amc.contract.renewed
     c. If payment fails:
        - Publish amc.contract.renewal_failed
        - Contract remains expiring for retry

3. Cron job runs daily, queries AmcContract where:
     status = 'expiring'
     AND endDate < now()
   Sets status = 'expired', expiredAt = now(), publishes amc.contract.expired
```

### Manual Renewal

```
POST /api/equipment/amc-contracts/:id/renew
  - Validates contract is in 'expiring' state (or 'expired' for late renewal)
  - Creates successor contract (same logic as auto-renewal but without payment gateway step)
  - Returns new contract
```

### Cancellation

```
POST /api/equipment/amc-contracts/:id/cancel
  - Validates contract is in 'active' or 'expiring' state
  - Requires reason
  - Sets status = 'cancelled', cancellationReason
  - Handles any outstanding claims (they remain valid for work done before cancellation)
  - Publishes amc.contract.cancelled
```

## 4. AMC-to-Equipment Relationship Model

### Design: Junction Table (`AmcContractEquipment`)

An AMC contract covers **one or more** equipment items. Equipment items can move between contracts (though this is discouraged mid-cycle).

| Aspect | Behaviour |
|--------|-----------|
| **Many-to-many** | A contract covers N items; an item may be covered by different contracts over time (but only one active contract at a time) |
| **Coverage period** | Each `AmcContractEquipment` row can have its own `coverageStartDate` / `coverageEndDate` (defaults to contract's start/end) |
| **Active constraint** | An equipment item cannot be covered by two active contracts simultaneously — enforced at the application layer |
| **Adding equipment mid-cycle** | Allowed via PATCH on the contract — requires reason for audit |
| **Removing equipment mid-cycle** | Allowed via soft-delete of the junction row — requires reason |

### Lookup Path

```
MaintenanceJob → amcContractId → AmcContract
MaintenanceJob → inventoryId → EquipmentInventory → AmcContractEquipment → AmcContract
```

Both paths are valid. `MaintenanceJob.amcContractId` is the primary assertion that a specific job is covered by a specific contract. When creating a job for AMC-covered equipment, the UI/service can auto-suggest the active contract based on `AmcContractEquipment` lookups.

### Guard: Active Contract Check

When a maintenance job is created with `type = 'amc'`:
- The specified `amcContractId` must reference an `active` AmcContract
- The equipment (`inventoryId`) must be linked to that contract via `AmcContractEquipment`
- If not linked, the service should reject with a clear error

## 5. Maintenance Claim Workflow

### Claim Lifecycle

```
submitted → under_review → approved → paid
                ↓
            rejected (terminal)
```

| State | Description |
|-------|-------------|
| `submitted` | Claim created when a covered maintenance job is completed |
| `under_review` | Claim being reviewed by admin/supplier |
| `approved` | Claim approved; `approvedAmount` set; ready for payment |
| `rejected` | Claim rejected; `rejectionReason` recorded (terminal) |
| `paid` | Claim paid; `paymentId` references billing.payments (terminal) |

### Transitions

| # | From | To | Trigger | Reason Required | Guard Condition |
|---|------|----|---------|-----------------|-----------------|
| 1 | — | submitted | Covered maintenance job completed | No | Job must have `amcContractId` and `amcCovered = true`; claim amount auto-populated from job costs |
| 2 | submitted | under_review | Admin begins review | No | — |
| 3 | under_review | approved | Admin approves claim | No | `approvedAmount` must be set; must not exceed `contract.claimAmountLimit` or period aggregate |
| 4 | under_review | rejected | Admin rejects claim | Yes | `rejectionReason` required |
| 5 | approved | paid | Payment processed | No | Creates/links `billing.payments` record; sets `paidAt` |

### Claim Auto-Creation

When a maintenance job with `amcContractId` and `type = 'amc'` transitions `in_progress → completed`:
- Service checks if `maintenanceJob.amcCovered = true`
- If yes, auto-creates an `AmcClaim` with:
  - `status = submitted`
  - `claimedAmount = maintenanceJob.totalCost`
  - `billingType = maintenanceJob.amcBillingType`
  - `claimNumber` = auto-generated (e.g., `AMC-CLM-{YYYYMMDD}-{sequence}`)

### Guard: Claim Limits

Before auto-creating or approving a claim:

| Guard | Behaviour |
|-------|-----------|
| `claimAmountLimit` | If `contract.claimAmountLimit` is set and `claimedAmount > claimAmountLimit`, claim is flagged for review (status = `submitted` but auto-approval blocked) |
| Per-period aggregate | Sum of all approved+paid claims in the current period; if adding this claim would exceed `claimAmountLimitPerPeriod`, flag for manual review |
| `claimLimitPerPeriod` | Count of approved claims in period; if exceeded, flag for manual review |

## 6. Payment Integration with `billing.payments`

### AMC Contract Payment (Contract Fee)

When an AMC contract is created or renewed, a payment may be collected:

| Scenario | Flow |
|----------|------|
| **New contract (one_time)** | Payment intent created for `contractValue`; on success, contract transitions `draft → active` |
| **New contract (installments)** | Payment schedule created; first installment collected immediately; subsequent installments handled via cron |
| **Auto-renewal** | Payment intent created for the next period's pro-rated value; on success, new contract created |
| **Manual renewal** | Admin can mark payment as received externally (via invoice reference) |

Payment records in `billing.payments`:
```
entityType = 'amc_contract'
entityId   = <AmcContract.id>
amount     = contractValue (or installment amount)
```

### AMC Claim Payment (Payout)

When a claim is approved, a payment/payout is processed to the supplier/provider:

| Scenario | Flow |
|----------|------|
| **Supplier payout** | Payment record created with `entityType = 'amc_claim'`; marks outgoing payment to supplier |
| **Adjustment/partial** | If `billingType = 'partial'` or `'excess'`, only the covered portion enters the claim; the excess is billed separately |

Payment records in `billing.payments`:
```
entityType = 'amc_claim'
entityId   = <AmcClaim.id>
amount     = approvedAmount
```

### Payment Entity Type Updates

Extend `createPaymentIntentSchema` entityType enum to include:
```typescript
entityType: z.enum(['membership', 'supplement_order', 'pt_session', 'equipment_lead', 'maintenance_job', 'amc_contract', 'amc_claim'])
```

### BillingType Logic

| `amcBillingType` | Meaning | Claim Amount |
|------------------|---------|-------------|
| `included` | Fully covered by AMC | `claimedAmount = totalCost` |
| `partial` | Partially covered; remainder billed separately | `claimedAmount = approvedAmount` (from contract terms) |
| `excess` | Entire cost is excess/out-of-pocket; AMC only facilitates the claim | `claimedAmount = 0`; `approvedAmount = 0`; job cost paid directly |

Every state transition records: `contractId`, `tenantId`, `fromStatus`, `toStatus`, `changedBy`, `reason`, `createdAt`.

## 10. Multi-Tenancy and RBAC Rules

### Multi-Tenancy

Same pattern as E4.2/E4.3:
- `tenantId` is denormalised on all three new models (`AmcContract`, `AmcContractEquipment`, `AmcClaim`) and both status log models
- All queries are scoped by `tenantId`
- Cross-tenant access returns 404 (not 403) to avoid information leakage

### RBAC

| Permission | Routes |
|------------|--------|
| `equipment:read` | GET all contract/claim endpoints |
| `equipment:manage` | POST, PATCH, DELETE all contract/claim endpoints; activate, renew, cancel, review, pay actions |

No new permissions are introduced — the existing `equipment:read` / `equipment:manage` pair covers the entire equipment domain (catalogue, inventory, maintenance, AMC).
