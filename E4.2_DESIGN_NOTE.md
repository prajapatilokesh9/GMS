# E4.2 — Equipment Inventory Design Note

## 1. Data Model

### EquipmentInventory

```prisma
model EquipmentInventory {
  id                 String    @id @default(uuid()) @db.Uuid
  tenantId           String    @map("tenant_id") @db.Uuid
  catalogueItemId    String    @map("catalogue_item_id") @db.Uuid
  serialNumber       String    @map("serial_number") @db.VarChar(200)
  gymId              String    @map("gym_id") @db.Uuid
  location           String?   @db.VarChar(200)       // within-gym zone/room/floor
  purchaseDate       DateTime  @map("purchase_date")
  purchaseCost       Decimal?  @map("purchase_cost") @db.Decimal(10, 2)
  supplier           String?   @db.VarChar(200)
  warrantyStartDate  DateTime? @map("warranty_start_date")
  warrantyEndDate    DateTime? @map("warranty_end_date")
  lastMaintenanceAt  DateTime? @map("last_maintenance_at")
  nextMaintenanceAt  DateTime? @map("next_maintenance_at")
  maintenanceIntervalMonths Int? @default(6) @map("maintenance_interval_months")
  status             String    @default("ordered") @db.VarChar(50)
  notes              String?   @db.Text
  isActive           Boolean   @default(true) @map("is_active")
  createdAt          DateTime  @default(now()) @map("created_at")
  updatedAt          DateTime  @updatedAt @map("updated_at")
  deletedAt          DateTime? @map("deleted_at")

  catalogueItem EquipmentCatalogue @relation(fields: [catalogueItemId], references: [id], onDelete: Restrict)
  gym           Gym                @relation(fields: [gymId], references: [id], onDelete: Restrict)
  tenant        Tenant             @relation(fields: [tenantId], references: [id], onDelete: Cascade)
  statusHistory InventoryStatusLog[]

  @@unique([tenantId, serialNumber])
  @@index([tenantId, gymId])
  @@index([tenantId, catalogueItemId])
  @@index([tenantId, status])
  @@index([tenantId, nextMaintenanceAt])
  @@index([tenantId, warrantyEndDate])
  @@map("equipment_inventory")
  @@schema("equipment")
}
```

### InventoryStatusLog

```prisma
model InventoryStatusLog {
  id          String   @id @default(uuid()) @db.Uuid
  tenantId    String   @map("tenant_id") @db.Uuid
  inventoryId String   @map("inventory_id") @db.Uuid
  fromStatus  String?  @map("from_status") @db.VarChar(50)
  toStatus    String   @map("to_status") @db.VarChar(50)
  changedBy   String?  @map("changed_by") @db.Uuid
  reason      String?  @map("reason") @db.VarChar(500)
  createdAt   DateTime @default(now()) @map("created_at")

  inventory EquipmentInventory @relation(fields: [inventoryId], references: [id], onDelete: Cascade)
  tenant    Tenant             @relation(fields: [tenantId], references: [id], onDelete: Cascade)

  @@index([inventoryId])
  @@index([tenantId])
  @@index([tenantId, createdAt])
  @@map("inventory_status_log")
  @@schema("equipment")
}
```

### Key Relationships

- `EquipmentInventory` → `EquipmentCatalogue` (M:1, Restrict delete — cannot remove a catalogue item that has inventory)
- `EquipmentInventory` → `Gym` (M:1, Restrict delete — gym must exist to hold equipment)
- `EquipmentInventory` → `InventoryStatusLog` (1:M, Cascade — status changes are immutable audit trail)
- `InventoryStatusLog` → `Tenant` (M:1, Cascade — enables direct tenant-scoped queries on status logs without JOIN)

### Schema

**equipment_inventory**

```
├── id                    UUID PK
├── tenant_id             UUID FK → tenant (Cascade)
├── catalogue_item_id     UUID FK → equipment_catalogue (Restrict)
├── serial_number         VARCHAR(200) — unique per tenant
├── gym_id                UUID FK → gym (Restrict)
├── location              VARCHAR(200) — optional zone/room
├── purchase_date         DATE
├── purchase_cost         DECIMAL(10,2) — nullable, actual cost paid
├── supplier              VARCHAR(200)
├── warranty_start_date   DATE — defaults to purchase_date
├── warranty_end_date     DATE — stored overridable value
├── last_maintenance_at   TIMESTAMP
├── next_maintenance_at   TIMESTAMP — computed: last + interval
├── maintenance_interval_months INT DEFAULT 6
├── status                VARCHAR(50) — state machine
├── notes                 TEXT
├── is_active             BOOLEAN DEFAULT true
├── created_at            TIMESTAMP
├── updated_at            TIMESTAMP
├── deleted_at            TIMESTAMP — soft-delete
└── UNIQUE(tenant_id, serial_number)
```

**inventory_status_log**

```
├── id                    UUID PK
├── tenant_id             UUID FK → tenant (Cascade) — denormalized for direct queryability
├── inventory_id          UUID FK → equipment_inventory (Cascade)
├── from_status           VARCHAR(50)
├── to_status             VARCHAR(50)
├── changed_by            UUID
├── reason                VARCHAR(500)
├── created_at            TIMESTAMP
└── INDEX(tenant_id, created_at)
```

### Default Values

| Field | Default Rule |
|-------|-------------|
| `warrantyStartDate` | `purchaseDate` |
| `warrantyEndDate` | `purchaseDate + catalogueItem.warrantyMonths` (overridable) |
| `maintenanceIntervalMonths` | `6` (overridable per item) |
| `nextMaintenanceAt` | `purchaseDate + maintenanceIntervalMonths` on create; `lastMaintenanceAt + interval` on maintenance |
| `status` | `'ordered'` |

## 2. Inventory Lifecycle State Machine

```
                ┌──────────┐
                │ ORDERED  │
                └────┬─────┘
                     │ delivered
                ┌────▼──────┐
                │ DELIVERED │
                └────┬──────┘
                     │ installed
                ┌────▼────┐
         ┌──────│ ACTIVE  │◄──────────────┐
         │      └────┬────┘               │
         │           │                    │
         │    ┌──────┼──────┐             │
         │    │      │      │             │
         │    ▼      ▼      ▼             │
         │  ┌───┐ ┌─────┐ ┌──────────┐   │
         │  │UND│ │ LOST│ │MAINTENANCE│───┘
         │  │REP│ │STOLN│ │ (scheduled│
         │  │AIR│ └──┬──┘ │  or unsch)│
         │  └───┘   │    └──────────┘
         │          ▼
         │    ┌──────────┐
         └────│ RECOVERED│ (found after loss)
              └──────────┘

    ┌───────────────────────┐
    │   DECOMMISSIONED      │ ◄── terminal from ACTIVE / MAINTENANCE / DAMAGED
    └───────────────────────┘
```

### Transition Table

| From | To | Trigger | Allowed By |
|------|----|---------|-----------|
| `ordered` | `delivered` | Goods receipt note | `equipment:manage` |
| `delivered` | `active` | Installation complete | `equipment:manage` |
| `ordered` | `active` | Direct deployment | `equipment:manage` |
| `active` | `maintenance` | Scheduled/unscheduled maintenance started | `equipment:manage` |
| `maintenance` | `active` | Maintenance completed, equipment cleared | `equipment:manage` |
| `active` | `lost_stolen` | Reported missing | `equipment:manage` |
| `lost_stolen` | `active` | Equipment recovered | `equipment:manage` |
| `active` | `decommissioned` | End of life / sold / scrapped | `equipment:manage` |
| `maintenance` | `decommissioned` | Beyond economical repair | `equipment:manage` |
| `lost_stolen` | `decommissioned` | Written off | `equipment:manage` |
| `delivered` | `decommissioned` | Never installed, disposed | `equipment:manage` |
| `ordered` | `decommissioned` | Order cancelled, never received | `equipment:manage` |
| `active` | `under_repair` | Damage reported (non-scheduled) | `equipment:manage` |
| `under_repair` | `active` | Repair complete | `equipment:manage` |
| `under_repair` | `decommissioned` | Beyond repair | `equipment:manage` |
| `damaged` | `under_repair` | Sent for repair | `equipment:manage` |
| `active` | `damaged` | Damage report filed | `equipment:manage` |
| `damaged` | `decommissioned` | Written off | `equipment:manage` |

### Guard Conditions

- Cannot transition from `lost_stolen` → `active` without an audit trail of recovery verification (`reason` must be ≥20 characters describing recovery details).
- All transitions record `fromStatus`, `toStatus`, `changedBy`, `reason` in `InventoryStatusLog`.
- Reason is **required** for all transitions marked "Yes" in the transition table.

> **Deferred to E4.3:** Guard `noActiveSessions` — rejecting decommission if item is assigned to an active workout. E4.2 does not implement session tracking; this guard will be added when E4.3 Maintenance introduces equipment-session linkage.

## 3. Serial Number Uniqueness Strategy

### Constraint

```
@@unique([tenantId, serialNumber])
```

### Rationale

1. **Tenant-scoped uniqueness**: Serial numbers are unique within a tenant's inventory. This prevents cross-tenant collisions while still enforcing uniqueness per organization.
2. **Manufacturer serials**: Manufacturers do issue globally unique serial numbers, but in practice duplicate entries (data entry errors) or re-used serials from returned equipment can occur. Tenant-scoped uniqueness catches intra-tenant duplicates regardless of source.
3. **Catalogue independence**: The same serial number cannot be used for two different catalogue items within the same tenant, preventing data entry mistakes where an operator tries to register the same physical asset twice.
4. **No global unique constraint**: Avoids `@unique` on `serialNumber` alone, which would prevent two different tenants from legitimately having the same manufacturer serial (e.g., two gym chains each buying a Treadmill Pro serial AB123 from FitTech).

### Edge Cases

| Scenario | Behavior |
|----------|----------|
| Soft-deleted item, same serial re-registered | **Blocked** — unique constraint still active on DB row. Soft-delete does not release uniqueness. |
| Bulk import with duplicate serials | **Rejected at first duplicate** — service attempts create in sequence, second hit returns 409. |
| Serial update to existing serial | **409 Conflict** — PATCH rejected if new serial collides with another item's serial (active or soft-deleted). |
| Manufacturer reuses serial across models | **Allowed** only if across different tenants; blocked within same tenant (same tenant = same manufacturer prefix likely). |

## 4. Warranty and Maintenance Scheduling Rules

### Warranty Calculation

```
warrantyStartDate = input.warrantyStartDate ?? purchaseDate
warrantyEndDate   = input.warrantyEndDate ?? purchaseDate + INTERVAL 'X months'
                    where X = catalogueItem.warrantyMonths ?? 12
```

- Both start and end dates are **overridable** on create/update to handle:
  - Second-hand equipment with partial warranty
  - Extended warranty purchased separately
  - Manufacturer warranty that started before delivery (e.g., warranty from invoice date)
- Warranty dates are **business dates** (no time component) — stored as `DateTime` but treated as dates.

### Maintenance Scheduling

```
nextMaintenanceAt = input.nextMaintenanceAt
                 ?? (lastMaintenanceAt ?? purchaseDate)
                    + INTERVAL 'Y months'
                    where Y = maintenanceIntervalMonths ?? 6
```

- **Default interval**: 6 months for all categories (configurable per item).
- **Category-based defaults**: Future enhancement — allow `equipment_category` table to define default intervals by category.
- **Scheduling**: `nextMaintenanceAt` is computed on create and after every maintenance completion.
- **Manual override**: `nextMaintenanceAt` can be explicitly set on PATCH.
- **Overdue detection**: Items where `nextMaintenanceAt < now() AND status = 'active'` are considered overdue for maintenance. A scheduled job (BullMQ worker) can identify and flag these.

### Reminder Rules (Future Scheduled Jobs)

| Event | Trigger | Offset |
|-------|---------|--------|
| `equipment.warranty.expiring_30d` | Warranty ends in 30 days | `warrantyEndDate - 30d` |
| `equipment.warranty.expired` | Warranty has ended | `warrantyEndDate + 1d` |
| `equipment.maintenance.due_14d` | Maintenance due in 14 days | `nextMaintenanceAt - 14d` |
| `equipment.maintenance.due_7d` | Maintenance due in 7 days | `nextMaintenanceAt - 7d` |
| `equipment.maintenance.overdue` | Maintenance past due | `nextMaintenanceAt + 1d` |
| `equipment.maintenance.overdue_30d` | Maintenance overdue by 30 days | `nextMaintenanceAt + 30d` |

These reminders are out of scope for the initial E4.2 delivery and will be added as part of E4.3 (Maintenance) or a post-E4.3 scheduled job sprint.

## 5. Gym Transfer Behavior

Equipment may need to be transferred between gyms (e.g., redistribution across locations, gym closure, sale of gym).

### Transfer Mechanism

Transfers are implemented as a `gymId` update via `PATCH /equipment/inventory/:id`. No special "transfer" endpoint is needed.

### Rules

| Aspect | Behavior |
|--------|----------|
| Status during transfer | Status **does not change** during a transfer. A working machine remains `active` after moving to a new gym. |
| Transfer condition | Equipment must be in `active` or `maintenance` status to be transferred. Items in `delivered`, `ordered`, `under_repair`, `damaged`, `lost_stolen`, or `decommissioned` status cannot be transferred. |
| Reason required | When `gymId` changes, `reason` is **required** in the PATCH body (min 10 chars describing the transfer reason). |
| Audit | A single audit `UPDATE` log entry captures both `oldValue: { gymId, gymName }` and `newValue: { gymId, gymName }`, plus the reason. |
| Event | `equipment.inventory.updated` is fired with `changedFields: ['gymId']` and old/new gym IDs. |
| Cross-tenant | Transfers across tenants are **not allowed**. A gym can only exist within the same tenant as the inventory item. |
| Status log | Gym transfer does **not** create an `InventoryStatusLog` entry — it is not a status transition. |

### Transfer State Machine Integration

```
active ──► active (gymId changes)
maintenance ──► maintenance (gymId changes, e.g., moved to a different gym for servicing)
```

All other statuses block transfer — the item must first return to `active` or be in `maintenance` before moving.

## 6. Audit and Event Matrix

### Audit Log Entries

| Action | Entity Type | Description | Triggered By |
|--------|-------------|-------------|-------------|
| `CREATE` | `equipment_inventory` | Item registered in inventory | POST `/equipment/inventory` |
| `UPDATE` | `equipment_inventory` | Fields updated (location, dates, etc.) | PATCH `/equipment/inventory/:id` |
| `STATUS_CHANGE` | `equipment_inventory` | Status transition | PATCH (status field) |
| `DELETE` | `equipment_inventory` | Soft-deleted | DELETE `/equipment/inventory/:id` |
| `RESTORE` | `equipment_inventory` | Restored from soft-delete | POST `/equipment/inventory/:id/restore` |

### Event Inventory

| Event Name | Queue | Payload | Trigger |
|------------|-------|---------|---------|
| `equipment.inventory.created` | `billing` | `inventoryId, catalogueId, serialNumber, gymId, status` | POST (create) |
| `equipment.inventory.updated` | `billing` | `inventoryId, changedFields[]` | PATCH (non-status) |
| `equipment.inventory.status.changed` | `billing` | `inventoryId, serialNumber, fromStatus, toStatus, reason` | PATCH (status transition) |
| `equipment.inventory.deactivated` | `billing` | `inventoryId, serialNumber, reason` | DELETE (soft-delete) |
| `equipment.inventory.restored` | `billing` | `inventoryId, serialNumber` | POST restore |

### Status Change Log (Immutable)

Every status transition is recorded in `InventoryStatusLog` with:
- `fromStatus` / `toStatus` — previous and new state
- `changedBy` — userId who performed the transition
- `reason` — mandatory free-text reason
- `createdAt` — immutable timestamp

The status log is append-only. No DELETE or UPDATE on status log entries.

### Integration Points

| System | Direction | Data |
|--------|-----------|------|
| E4.1 Catalogue | Read | `catalogueItem` reference, `warrantyMonths`, specs |
| Gym module | Read | `gymId` reference, location context |
| E4.3 Maintenance (future) | Write/Read | Maintenance events reference `inventoryId`, update `lastMaintenanceAt`/`nextMaintenanceAt` |
| E4.4 AMC (future) | Write/Read | AMC contracts tied to inventory items |
| Scheduled jobs (future) | Trigger | Warranty expiry and maintenance reminders |
